# pharos 概览

> 讲 pharos 怎么运作、为什么这么设计(理解导向)。要上手用 → [README](../README.md);要改代码 → [architecture](architecture.md)。本文不讲安装、不列字段。

## 它解决什么

一台开发机上跑着十几个易碎的本地服务——SSH 隧道、本地推理服务、launchd 任务、后台 daemon。它们会无声地挂掉,而你往往用到时才发现。pharos 是这台机器的灯塔:周期性确认每盏灯还亮着,灭了就示警。

它**只监控 + 示警**,不重启、不修、不存历史曲线——那些是别的工具的事。这条边界是有意的:一个会自动"修"东西的监控,本身就成了一个会出错的变更源。

## 三个概念

**检查(check)** = 一次对某目标的只读探测,产出一个判定(`OK` / `DEGRADED` / `DOWN` / `UNKNOWN`)。关键设计:检查是**参数化的类型 + 配置**,不是代码。"查某个 launchd 任务在不在"是一个通用类型;"查 `com.example.tunnel` 在不在"是你 `checks.toml` 里的一行。**为什么**:这样框架对所有人通用,而每个人的监控只是数据。真正没法参数化的特殊状态语义才写成代码,且明确隔离为 contrib,不污染通用核心。

**告警 = 工单(ticket),不是推送(push)**。这是 pharos 最核心的取舍。一条 fire-and-forget 的推送发完就没了;但真正处理告警的是人 / agent,需要一个**可查询、有状态**的视图:现在几个在烧、哪个在查、哪个已处理。所以告警有生命周期——`firing → acked → resolved`,像工单系统。**好处**:去重和静音不再需要额外机制,它们是状态的自然结果——已在烧的工单不重复通知(去重),标了"在处理"的不再吵(静音)。通知渠道退回它该有的角色:**状态翻转时通知人**,而不是每轮刷屏。

**通知 = 可插拔渠道 + 按来源路由**。一个告警往哪发,取决于它的 `source`:基础设施告警发到一个渠道,某个工作流发到另一个。渠道本身(`console` / `dingtalk` / 将来的 Slack 等)是可插拔后端。**为什么**:让别人能接自己的传输方式,也让一台机器的不同告警去到不同的人。

## 一次 run 怎么流起来

把三个概念串成一条线(这就是 pharos 整个怎么运作):

1. 加载 `checks.toml`,跑出一组**判定**。
2. 拿这组判定和**工单库**对账(reconcile):新失败的开工单,仍失败的只更新,**这次跑了且现在恢复的**关掉。
3. 对"新开"和"关闭"两种翻转发**通知**(按来源路由);其余沉默。

第 2 步的对账藏着关键不变量:**恢复 = 这次真的跑了且判定为 OK**。一个检查这次没跑到(比如你只跑了子集),或判定为 `UNKNOWN`(测不出),都**不算恢复**——否则一次局部运行就会无声地清掉真实在烧的告警。"没看到"不等于"没事"。

## 灯塔类比的边界

灯塔是个好心智模型(守望 + 示警),但别推太远:pharos 不引航(不告诉你怎么修),也不照向远方(不主动巡检远程系统,只查这台机器能触达的东西)。它是**一个固定点,确认本地的灯还亮着,灭了拉响警报**——仅此。

## 代价

这套设计的代价值得点明:工单状态存在本地 sqlite,所以它是**单机**视角,不聚合多台;检查是**点状**的(每几分钟一次),不是连续监控,短于间隔的抖动它看不见。对一台个人开发机,这些取舍是对的;要监控机群或要秒级精度,这不是合适的工具。
